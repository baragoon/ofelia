Interpretation:

- `45` successful executions matches three valid jobs running across roughly fifteen scheduler cycles
- `15` wrong-host failures confirms the `pin-mismatch` job was routed to `dind3` and could not see `pin-target-dind1` on `dind1`
- the three valid jobs did not produce container lookup failures, which is consistent with correct host resolution and successful remote exec creation

## Important caveat

A stronger validation attempt wrote commands like `sh -lc "echo pinned-dind1 >> /tmp/ofelia.log"` into the target containers, but those file writes did not materialize.

This is explained by current command handling in `core/execjob.go`, which passes `args.GetArgs(j.Command)` directly to Docker exec. That tokenizes shell syntax into argv components, so quoted shell fragments such as `sh -lc "... >> file"` are not preserved as a single shell command string.

This does not invalidate the multi-host result. The host-selection evidence still holds because:

- the three correct jobs consistently succeeded
- the wrong-host job consistently failed with `No such container`

## Conclusion

The three-host local integration test passed for the feature under test:

- Ofelia connected successfully to three local TLS-enabled Docker daemons using env-based host configuration
- self-signed TLS certificates worked correctly for all three daemons
- pinned jobs were routed to the intended hosts
- a deliberately misrouted job failed exactly as expected on the wrong host

## Follow-up

If stronger container-side assertions are needed, add a command form that survives argv tokenization, or change exec command handling to preserve shell command grouping for cases like `sh -c '...'`.
